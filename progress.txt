KUA-LOGIC-001 - 2026-05-13 - Built calculateKua as a pure function with the digital-root reduction and the Kua 5 reassignment baked in. The non-obvious correctness moment: when a male calculation hits subtractFrom - reduced = 0 (which can happen post-2000 if reduced = 9), the result must wrap to 9, not be left at 0. The +9 wrap is in place. Also confirmed the female reassignment fires when reduceToSingleDigit(reduced + addend) lands on 5, not when (reduced + addend) is literally 5: female 1990 reduces last-two-digits 90 to 9, then 9+5=14 reduces to 5, then reassigns to 8. The chapter 6 phrasing 'if the result is 5' was ambiguous about whether reassignment happens before or after the second reduction; the worked examples settle it (after).

UI-001 - 2026-05-13 - The biggest task; landed in three files (index.html template, ui.js, main.css). Decisions worth recording: (1) The radio buttons are styled label-as-button pads with the native input visually hidden but kept in the DOM for accessibility, then revealed via :has(input:checked) for the active state. This pattern gives a 48px touch target instead of the ~16px native radio circle and looks calm with the editorial aesthetic. (2) The result card uses a <template> element cloned at render time, with [data-bind="..."] attributes acting as the slot interface. This keeps the markup declarative and means SEO-001 can later edit copy inside the template without touching JS. (3) The reduced-motion media query disables the cardRise animation and the smooth scroll. (4) The result card has a 4px gradient bar across the top (terracotta to sage) - a single editorial touch that signals "this is a finished thing" without adding noise. The frontend-design skill's instruction to commit to one memorable detail rather than scatter micro-interactions justified that call.

PRINT-001 - 2026-05-13 - Print rules wrapped in @media print despite the link element already using media='print' - belt and braces, and lets the file be loaded unconditionally in the future without breaking screen view. The key accessibility consideration: favourable vs unfavourable must be readable in greyscale, since most home printers print B&W by default. Solved by three redundant cues (the ★ vs × icon, the solid vs dashed left border, and the 'Favourable' / 'Avoid' text), so colourblind readers and B&W printers both work. Used mm units for print dimensions (consistent with A4 conventions) and pt for type (consistent with print typography).

SEO-001 - 2026-05-13 - The marketing copywriting skill and the ai-seo skill both pushed in the same direction: lead with the direct answer, keep answer blocks to 40-60 words, match heading wording to query phrasing. CNY-001 - 2026-05-13 - User flagged a correctness concern: the original form took only birth year, relying on a helper hint that told users to manually subtract one year if born before Chinese New Year. That's what chapter 6 actually documents, and it matches most online Kua calculators, but it's high-friction and error-prone. Upgraded the form to year + month + day, added a CNY-date lookup table for 1900-2050 in cny.js, and added effectiveKuaYear(year, month, day) which returns the adjusted year and a metadata object describing whether an adjustment happened. The result card now shows a sage-tinted notice when an adjustment fires ('Your birth date is before Chinese New Year (Feb 9, 1986), so the Kua formula uses 1985 as your Chinese solar-calendar year, not 1986'). Verified the worked examples still work: male Jul 1978 -> Kua 4, female Aug 1985 -> Kua 9. Verified the new edge case: female Jan 15, 1986 (before CNY Feb 9, 1986) -> kuaYear=1985 -> Kua 9. The change also closes a real correctness gap because users who didn't read the helper text were silently getting the wrong Kua. Added a new FAQ entry covering CNY handling, both in the visible page and the FAQPage JSON-LD.
